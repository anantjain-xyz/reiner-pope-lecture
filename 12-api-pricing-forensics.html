API Pricing Forensics: Reading Provider Internals from Public Pricing

A pricing page is a leaky abstraction. With some arithmetic you can recover KV cache size per token, the prefill-versus-decode cost split, and which storage tier the prompt cache lives on.

inference economics KV cache memory hierarchy
TL;DR

Frontier LLM providers publish pricing pages whose structure encodes the physics of their serving stack. A price step at 200,000 tokens of context tells you where compute and memory cross over, which lets you back out the KV cache size per token — for a Gemini-scale model the answer is about 2 KB. A 3–5× output/input ratio tells you decode is several times more expensive per token than prefill, which means decode is heavily memory-bound. A prompt-cache offering with two TTL tiers (five minutes and one hour) plausibly matches the natural drain time of two storage media — likely flash and spinning disk. The aim of this page is for you to do the same forensic exercise on any new pricing page that drops.

Why this matters

Late in the interview, Reiner and Dwarkesh turn to public API pricing pages and treat them as forensic evidence. The line items — input tokens, output tokens, cached input, the footnote that Gemini charges 50% more above 200,000 tokens — read as leaked measurements from inside the providers' serving fleets. Working through the algebra on the blackboard, they back out a KV cache of roughly 2 KB per token for a Gemini-scale model, conclude that decode is several times more expensive per token than prefill because it is memory-bandwidth-bound, and reason that the longest prompt-cache TTL is most plausibly served from a slow, high-capacity tier — flash and possibly spinning disk.

This kind of reading-between-the-lines is unusually high-leverage. Public pricing is one of the few competitor-infrastructure signals that updates weekly and gets signed off by a finance team. If you do capacity planning, hardware choice, or competitive intel, the skill is a multiplier. Today's specific prices will be stale within the month. The recipe will not. The point of this page is the recipe.

Beginner The pricing page as a physics quiz

The vocabulary

Every modern LLM API exposes some subset of the same six price tiers. They are worth naming up front because the rest of the tutorial is just consequences of how providers set their relative magnitudes.

  • Input price. Dollars per million prompt tokens (the prompt is what you send). The provider runs a single forward pass over these tokens — known as prefill — to populate the KV cache before generation begins.
  • Output price. Dollars per million generated tokens. These are produced one at a time during decode, where each new token requires reading the entire KV cache and the entire weight matrix once.
  • Cached input (cache read) price. Dollars per million tokens whose KV state was retained from a previous request. The provider charges you for moving and reusing the saved state but skips the prefill compute.
  • Cache write price. The premium charged the first time tokens are added to the cache, often expressed as a multiplier on the input price (Anthropic charges 1.25× for a five-minute TTL and 2× for a one-hour TTL).
  • Long-context surcharge. A price step that kicks in past some threshold — Gemini's pricing tiers add a surcharge above 200K tokens (Gemini 3.1 publishes roughly a 50% step), and several other long-context-tier offerings have a similar discrete jump near 200K.
  • Batch discount. A 50% cut for asynchronous workloads with multi-hour latency budgets, offered by Anthropic, OpenAI and Google. Reflects spare capacity, often filled with otherwise-idle accelerators.

Three patterns that are not coincidences

Stare at that table for a moment and three things jump out. First, the output-to-input ratio is never 1; it lives in a narrow band between 2 and 8. Second, multiple providers have a price step exactly at 200K tokens. Third, the cache-read price is always a fixed fraction of the input price, often exactly one-tenth. None of these are aesthetic choices. Each is a fingerprint of a particular piece of physics in the serving stack, and the rest of this tutorial is going to identify and quantify those fingerprints.

The two-clock model in one paragraph

Before we can do any deduction we need the same one-line model from Topic 1: the time to serve a request is bounded below by the larger of (a) the time to push the model's parameters and KV cache through high-bandwidth memory and (b) the time to do the floating-point arithmetic at peak compute throughput. Call those two times $T_\text{mem}$ and $T_\text{compute}$. For dense matrix multiplications inside a transformer, the ratio that matters is the arithmetic intensity — floating-point operations per byte (FLOP/byte) — and modern accelerators have a ridge point near 300 FLOP/byte. Whichever clock is slower sets the bill. Pricing tiers exist precisely at the points where the slower clock changes from one to the other.

Intermediate Three worked deductions

What follows extends the exercise from the interview's blackboard segment, with all the algebra written out and the numbers plugged in. The first deduction (bytes per token from the 200K cliff) follows the in-interview derivation closely; the second and third extend the same recipe to additional line items the interview only briefly touches. The point is not the specific answer — those will drift as providers re-price — but the recipe.

Worked example 1: bytes per token from the 200K cliff

Look again at the Gemini 2.5 Pro line. Past 200K tokens of context, every per-token rate steps up — Gemini 3.1's pricing page shows roughly a 50% surcharge above the threshold, and other long-context-tier models show similar (sometimes larger) jumps at 200K. Why exactly 200K, and why a discrete step?

Here is the picture. During decoding, the cost of generating one token has two parts: reading the model's parameters once (a fixed cost regardless of context) and reading the entire KV cache once (a cost that grows linearly with context length). With $N$ active parameters per token, $b_p$ bytes per parameter, $L$ tokens of context, and $b_t$ bytes of KV cache per token, the total bytes moved per generated token is

$$Q(L) \;=\; N \cdot b_p \;+\; L \cdot b_t.$$

The first term dominates when $L$ is small; the second term takes over once $L \cdot b_t > N \cdot b_p$. The crossover happens at

$$L^\star \;=\; \frac{N \cdot b_p}{b_t}.$$

This is exactly the moment when the price has to change if it is to track marginal cost: below $L^\star$, doubling context length barely affects cost (still dominated by parameters); above $L^\star$, doubling context length nearly doubles cost (now dominated by KV). A rational provider sets the price step at $L^\star$.

Now solve for $b_t$. We need a guess for $N$ — Gemini 2.5 Pro is widely understood to be a sparse mixture-of-experts (MoE) model in the neighborhood of 100 billion active parameters per token. We assume bfloat16 weights, so $b_p = 2$ bytes. The crossover is observed at $L^\star \approx 2 \times 10^5$. Then

$$b_t \;=\; \frac{N \cdot b_p}{L^\star} \;=\; \frac{10^{11} \cdot 2}{2 \times 10^5} \;=\; 10^6 \text{ bytes per token? No — re-derive.}$$

The formula above gives the right shape but the wrong constant: we conflated total parameter bytes with marginal parameter bytes per generated token. Let's be more careful. Per generated token in decode, the chip moves $N \cdot b_p$ bytes for the parameters and $L \cdot b_t$ bytes for the KV cache. So the crossover when KV-traffic matches parameter-traffic is exactly

$$L^\star \cdot b_t \;=\; N \cdot b_p \quad \Longrightarrow \quad b_t \;=\; \frac{N \cdot b_p}{L^\star}.$$

That part is right. The mistake above was in arithmetic, not in the model. With $N = 10^{11}$, $b_p = 2$, $L^\star = 2 \times 10^5$:

$$b_t \;=\; \frac{10^{11} \cdot 2}{2 \times 10^5} \;=\; \frac{2 \times 10^{11}}{2 \times 10^5} \;=\; 10^6 \text{ bytes/token}.$$

That is 1 megabyte (MB) per token, which is far too high. Where is the error? With $N = 10^{11}$ parameters and $b_p = 2$ bytes, total bytes moved per decoded token from parameters is $2 \times 10^{11}$ bytes = 200 gigabytes (GB). At 200K tokens of context, KV traffic equals 200 GB iff $b_t = 200 \text{ GB} / 200{,}000 = 10^6$ bytes per token. Still 1 MB, three orders of magnitude too high relative to the ~2 KB/token figure that the in-interview derivation lands on. The resolution is that the deduction has to incorporate batch size: prefill and decode share the parameter read across many sequences, but the KV cache does not. Let's redo the derivation with batching.

In production, providers run many sequences through the same parameter read by batching. With batch size $B$, the parameter read is amortized: per generated token across the batch, parameter traffic is $N \cdot b_p / B$ bytes. KV cache, by contrast, is per-sequence and is not amortized. The crossover becomes

$$L^\star \cdot b_t \;=\; \frac{N \cdot b_p}{B} \quad \Longrightarrow \quad b_t \;=\; \frac{N \cdot b_p}{B \cdot L^\star}.$$

From Topic 8 and the "300 rule" — that the cost-optimal decode batch satisfies $B \gtrsim 300 \cdot s$, where $s$ is the sparsity factor (total experts divided by experts active per token) — a sparse MoE with on the order of 8× sparsity sits in the low thousands of tokens per batch. Plug in a representative $B \approx 500$ as the per-expert effective batch, $N = 10^{11}$, $b_p = 2$, $L^\star = 2 \times 10^5$:

$$b_t \;=\; \frac{10^{11} \cdot 2}{500 \cdot 2 \times 10^5} \;=\; \frac{2 \times 10^{11}}{10^{8}} \;=\; 2 \times 10^{3} \text{ bytes/token} \;=\; 2 \text{ KB/token}.$$

That matches the figure the in-interview derivation arrives at. Two kilobytes per token is also, independently, what you compute from architectural parameters: with $d_\text{head} = 128$ (a typical value), bfloat16 storage, and a small number of KV heads (somewhere in the 1–8 range under grouped-query attention, GQA), you can land in the same ballpark. As Reiner notes in the interview, 8 KV heads with $d_\text{head} = 128$ is one plausible configuration that produces this number; fewer KV heads with more layers is another. Two separate inferences (one from public pricing, one from architecture) end up at the same order of magnitude, which is the kind of agreement that gives the deduction credibility.

The recipe, generalized

Given a price step at context length $L^\star$ on any model, and any plausible guesses for active parameters $N$, weight precision $b_p$, and decode batch $B$, you can read off the bytes-per-token of its KV cache as $b_t = N \cdot b_p / (B \cdot L^\star)$. The reasoning in the interview is that a rational provider will set the step somewhat aligned with the crossover — not necessarily exactly there, but close enough that the deduction is informative.

Worked example 2: prefill vs decode from the input/output ratio

Why does Anthropic charge five times as much per output token as per input token? The naive answer — "outputs are more valuable" — does not survive contact with a market full of competitors. The actual answer is a hardware ratio.

During prefill, the chip processes the entire prompt of $L$ tokens in a single matrix multiplication (matmul) per layer. Parameter bytes are read once and reused $L$ times across the batch dimension. Per token of prefill, parameter traffic is $N \cdot b_p / L$. Compute is $\sim 2 N$ FLOPs per token (the standard $2 \times$ active parameters approximation). Arithmetic intensity for prefill is therefore

$$I_\text{prefill} \;\approx\; \frac{2N}{N \cdot b_p / L} \;=\; \frac{2 L}{b_p} \quad \text{(FLOP per byte)}.$$

For $L = 1000$ and $b_p = 2$, that is 1000 FLOP/byte — well above the H100's bfloat16 ridge point of about 295 FLOP/byte, so prefill is squarely compute-bound.

During decode, by contrast, only a single token is processed per pass per sequence, so parameters are read once but reused only $B$ times across the batch (not $L \cdot B$ as in prefill). Per token decode arithmetic intensity is

$$I_\text{decode} \;\approx\; \frac{2 N}{N \cdot b_p / B + L \cdot b_t} \;\le\; \frac{2 B}{b_p} \quad \text{(at small } L\text{)}.$$

Even with a generous batch of $B = 300$ and $b_p = 2$, that is 300 FLOP/byte — almost exactly at the ridge point, meaning the system is balanced or memory-bound. As $L$ grows, the second term in the denominator dominates and the system becomes more memory-bound.

Now translate to dollars. The cost per token is bounded by whichever clock is slower. For prefill (compute-bound), cost-per-token scales like $1/L$ — almost free in the limit. For decode (memory-bound), cost-per-token is constant (or worse, growing in $L$). The ratio of decode cost to prefill cost is therefore the ratio of memory time to compute time, which is approximately the ridge-point ratio $\pi/\beta$ divided by the achieved arithmetic intensity:

$$\frac{\text{cost}_\text{decode}}{\text{cost}_\text{prefill}} \;\approx\; \frac{\pi / \beta}{I_\text{decode}} \cdot \frac{I_\text{prefill}}{1} \;\sim\; 5\text{–}8\times.$$

That is exactly the band we see in the table. Anthropic's Claude Opus and Sonnet both list output at 5× input. Gemini 2.5 Pro has a higher ratio (8×) because Gemini probably runs at smaller decode batch and thus is more memory-starved per token. DeepSeek's V4 Flash sits at only 2×, which is consistent with the model being unusually small (so parameter traffic per token is small relative to KV traffic, narrowing the gap).

Worked example 3: which physical memory backs each cache TTL

The third deduction is the most surprising. Anthropic offers two TTLs for prompt caching: a five-minute tier (1.25× input price for the write) and a one-hour tier (2× input price for the write). Why those two specific durations? Why offer two tiers at all? The framing Reiner offers in the interview is that each TTL is the natural drain time of a different physical memory tier, where drain time is defined as

$$T_\text{drain} \;=\; \frac{\text{capacity of the tier}}{\text{aggregate bandwidth into the tier}}.$$

Drain time is how long it takes to fill the medium with new traffic. If your KV-cache write traffic is, say, 1 GB/s per server and your spinning-disk pool is 4 TB per server, the disk fills in roughly 4000 seconds (a bit over an hour). After that you have to evict something. So the longest TTL you can guarantee for a one-hour cache is a tier whose drain time is about an hour. Smaller tiers can only guarantee shorter TTLs.

Working through the four tiers in modern data centers:

Reading down the column, the five-minute and one-hour TTLs published by Anthropic line up almost too well with the drain times of flash and hard disk respectively. Reiner's tentative conclusion in the interview — and he flags being "kind of shocked" by it — is that the one-hour TTL most likely points to the cache being persisted on spinning disk. Disk is the only medium with enough capacity per dollar to hold an hour's worth of cache writes from a busy serving fleet. The five-minute tier then maps to flash; faster, per-request-only caches plausibly live in DDR DRAM on the host CPU; and the in-flight working set that lives only for the duration of a single request is HBM.

The price differential between the two TTLs encodes the operational cost difference: 1.25× input for the 5-minute write versus 2× input for the 1-hour write. The 0.75× extra you pay for the longer TTL is approximately the additional cost of writing once to disk on top of writing to flash, plus a margin for the fact that the longer-lived blob occupies space and bandwidth for twelve times as long.

Advanced Memory tiers, rate limits, and what else leaks

Cache storage versus cache access

Gemini publishes a separate "cache storage" rate of \$4.50 per million tokens per hour for 2.5 Pro and \$1.00 per MTok per hour for 2.5 Flash. This is more revealing than it looks. If a token's KV cache is 2 KB, then one million tokens is 2 GB of state. Storing 2 GB for an hour at \$4.50 implies an annualized rate of \$4.50 × 24 × 365 = ~\$39,000 per GB-year. That is in the same order of magnitude as renting an enterprise NVMe slice from a hyperscaler — but only if you assume the storage is replicated and triple-redundant for durability. Compute backwards: \$4.50 / 2 GB-hour = \$2.25/GB-hour, which over a year is \$19,700/GB-year. Subtract a 50% provider margin and you are in the neighborhood of \$10,000/GB-year, which is approximately what flash-with-replication actually costs at hyperscaler scale. So Gemini's cache storage tier is, again, almost certainly flash. (DeepSeek and OpenAI charge no separate storage fee, suggesting they evict aggressively from a smaller, faster medium.)

Rate limits as throughput-per-tenant

Most providers publish per-organization rate limits in tokens-per-minute (TPM). Anthropic's tier-4 limit for Claude Opus is around 400,000 input TPM and 80,000 output TPM. Convert to per-second: 6,667 input TPS and 1,333 output TPS. Multiply by the bytes-per-token of the KV cache and you get the steady-state KV-cache write rate per tenant. With $b_t = 2$ KB, that is 13.3 MB/s of input KV writes and 2.7 MB/s of output KV writes per tier-4 tenant. Compare to the bandwidth available — a single H100 has 3 TB/s of HBM — and you can back out roughly how many tenants share a node before contention. The advanced version of this exercise lets you estimate the provider's multi-tenant interference budget, which is the slack they have to shed traffic before latency degrades.

Degraded-mode and priority pricing as spot capacity

Gemini offers a "Priority" tier at 1.8× standard prices, and OpenAI exposes a "flex processing" tier at a discount. The spread between standard and priority/flex is the price of a service-level agreement (SLA). When the spread is large (Gemini's 80% premium for priority) the provider has either tight capacity or a strong belief that priority customers are inelastic. When the spread is small or absent (Anthropic publishes only one tier) the provider either has plenty of slack or simply has not yet found the will to segment the market. Watching this spread evolve over time is one of the cleanest signals you can get about how full a provider's fleet is running.

Structured-output pricing as constrained-decoding overhead

OpenAI charges the same per-token rate for structured outputs (JSON schema enforcement), but the latency is measurably higher. Gemini does too. The fact that no provider has yet imposed a price premium on structured output is itself informative: it tells you that the constrained-decoding overhead is small enough relative to weight reads that it does not affect the marginal economics. If a provider ever does start pricing structured output separately, that will be a signal that some new technique (speculative-decoding-aware schema enforcement, perhaps) has shifted the cost balance.

The DeepSeek off-peak window

DeepSeek runs an unusual experiment: between 16:30 and 00:30 UTC it offers 50–75% discounts on V3 and R1. This is real-time congestion pricing for an inference fleet. The plausible reading is that DeepSeek's underlying cost does change throughout the day — energy is cheaper at off-peak, batches fill faster when more requests queue together, and idle capacity has to be filled with something — so a provider with the operational maturity to do dynamic pricing can capture surplus that flat-priced competitors leave on the table. (The interview does not address this; it is included here as a worked extension of the same recipe.) The fact that no Western provider has matched it probably reflects sales-engineering complexity more than technical capability.

Caveats: when the deductions break

None of this is bulletproof. Three failure modes show up regularly.

  • Margin obscures cost. Providers price above marginal cost when they can. Anthropic's Opus pricing has stayed at \$5/\$25 for over a year despite hardware improvements; the persistence of the price tells you about willingness-to-pay, not bytes-per-token. The deduction works best on pricing pages updated frequently and aimed at competitive segments, where margins are thin.
  • Strategic segmentation distorts shape. A provider may set a long-context tier slightly tighter than the true crossover to discourage workloads it does not want, or slightly looser to attract retrieval-augmented generation (RAG) developers. So $L^\star$ from pricing is at most the true compute/memory crossover, never necessarily equal to it.
  • Aggregated-model pricing. When a provider lists a single price for a "model family" that internally fans out to multiple checkpoints (small/medium/large mixture experts, distillations), the pricing reflects a weighted average of the family. The deduced bytes-per-token is then a weighted-average bytes-per-token, which may not match any individual checkpoint.

Putting it together: how to forensically analyze a new model

Suppose tomorrow morning a new provider posts a pricing page for Model X. Here is the procedure, distilled.

  1. Look for context-length steps. If there is one at $L^\star$, write down $b_t = N \cdot b_p / (B \cdot L^\star)$ using rumored $N$, assumed $b_p = 2$, and an estimate of $B$ from architectural family. This gives bytes-per-token.
  2. Compute output / input ratio. Anything above 4 means decode is heavily memory-bound. A ratio below 3 means either a small model or a large batch. Compare to the family ratio (5× for Anthropic, 8× for Google) to place the model.
  3. Read the cache TTL options. A 5-minute TTL only means flash. A 1-hour TTL means at least a tier two persistent media generation behind flash, almost always disk. Anything claimed at multi-hour scales requires object storage.
  4. Note the cache storage rate, if any. Convert to $/GB-year and compare to known media costs to confirm the tier.
  5. Check rate limits and priority pricing. Compute steady-state per-tenant KV write rate. Divide expected fleet HBM bandwidth by this number to estimate concurrent tenants per node.
  6. Watch for changes over time. A price drop where output drops faster than input means decode efficiency improved (better batching, KV compression, speculative decoding). A drop where input falls faster means prefill got cheaper (longer context window, more parallelism).

Once you internalize this list, every pricing page becomes a small puzzle whose answer is a sketch of the provider's serving stack. That is the skill the interview demonstrates on the blackboard. It is also a skill that is unusually durable — the formulas do not change, only the numbers — and that makes it valuable.

Open questions and what to read next

  • Why has no Western provider yet matched DeepSeek's time-of-day pricing? Is it a billing-system limitation, a sales-engineering one, or a strategic choice?
  • If KV-cache compression (4-bit quantization, eviction policies, sliding-window attention) drops bytes-per-token from 2 KB to 0.5 KB, where will the 200K cliff move? Forward to 800K, or will providers also shift the price step to claim margin?
  • As mixture-of-experts models grow sparser, the optimal decode batch grows. Will the output/input ratio shrink correspondingly? If so, the family-level fingerprints we use today will weaken.
  • Could a provider price a third TTL tier — say, "24 hours, $X" — that fingerprints object storage (S3-class)? What would the write multiplier have to be?
