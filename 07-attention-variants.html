Attention Variants: MHA, MQA, GQA, MLA

The shape of the attention block sets your dollars-per-million-tokens. A decade of trimming the KV cache, in one page.

attention KV cache inference DeepSeek
TL;DR

Standard Multi-Head Attention (MHA) stores one key and one value per head per token. The cache balloons. The field has spent five years trimming it. Multi-Query Attention (MQA) shares a single key/value across all query heads. Grouped-Query Attention (GQA) compromises by sharing each key/value across a group of query heads (Llama 2/3, Mistral, most of today's open models). Multi-Head Latent Attention (MLA), introduced in DeepSeek-V2, ditches the per-head KV idea entirely and caches one low-rank latent vector per token, expanding it on the fly. The Pareto frontier of cache size vs. quality has moved from MHA's hundreds of KB per token down to MLA's 70 bytes per layer per token. That one number is most of what determines how cheap your model is to serve at long context.

Why this matters

Halfway through the interview, Reiner works out the size of Gemini's KV cache from API pricing alone. The price doubles past 200K tokens. That tells him the cache crosses the cost of the weights at roughly that length, which back-solves to about 2 KB per token. Then he makes a distinction that trips up most people reading transformer code for the first time:

"There are KV heads — those are the heads stored in memory, carrying the past tokens' content. And there are Q heads, the retrieval heads used by the token that's currently attending. The number of KV heads is typically 1 to 8, much smaller than the number of query heads."

A decade of architecture work sits behind that sentence. Once you know why those two counts get sized independently, and what DeepSeek did to shrink the KV side further, the attention sections of any modern model report read fluently. You can predict serving cost from a config file, and you can see why an architecture that wins benchmarks can lose in production.

This page builds that picture from scratch. The goal is for you to look at a Hugging Face config.json, identify the attention variant, compute its bytes per token, and explain the trade-off the architects made.

Beginner Self-attention and the KV cache

The 2017 Vaswani recap

In the original transformer, each token $x_t$ produces three vectors per attention head: a query $q_t$, a key $k_t$, and a value $v_t$. The attention output for the token at position $t$ is

$\displaystyle o_t = \sum_{s \le t} \mathrm{softmax}\!\left(\frac{q_t \cdot k_s}{\sqrt{d_\text{head}}}\right) v_s.$

Read this as: the current token's query "asks a question," every previous token's key "answers" with how relevant it is, and the value carries the actual information that gets averaged in. With $h$ heads of dimension $d_\text{head}$, the model runs $h$ copies of this in parallel and concatenates the outputs. Different heads end up specialising — some look at the previous token, some at the start of a sentence, some at coreferent nouns.

Why we cache K and V (but not Q)

At inference time, generating one token means computing $q_t, k_t, v_t$ for the new position and computing attention against all previous keys and values. If we recomputed those from scratch every step, decoding would be $O(n^2)$ per token. Instead, we cache $k_s$ and $v_s$ for every previous position $s$ and reuse them. The query is fresh for each new token, so it does not need to be cached.

This is the KV cache. It is the single biggest piece of inference state, and its size grows linearly with both context length and batch size. For long-context, large-batch serving — which is most production serving — the KV cache, not the model weights, dominates GPU memory.

The bytes-per-token formula

For Multi-Head Attention with $L$ layers, $h$ heads of dimension $d_\text{head}$, and $b$ bytes per element (2 for FP16/BF16, 1 for FP8):

$\displaystyle \text{bytes per token} = 2 \times L \times h \times d_\text{head} \times b.$

The leading 2 is because we store both K and V. Plug in Llama-1 65B: 80 layers, 64 heads, 128-dim heads, FP16. That gives $2 \times 80 \times 64 \times 128 \times 2 = 2{,}621{,}440$ bytes — 2.5 MB per token. A single 32K-context request consumes 80 GB of KV cache: more than an H100 holds. This is why the Vaswani-era attention does not survive into modern long-context serving.

Worked example: Llama-1 65B

$L=80$, $h=64$, $d_\text{head}=128$, FP16 → 2.62 MB / token. For comparison, a 8K-context, batch-size-32 request would need 670 GB of KV memory. The fix the field reached for, between 2019 and 2024, is the subject of the rest of this tutorial.

The retrieval intuition

The Pope framing of "Q heads = retrievers, KV heads = stored content" is the right mental model for everything below. Think of the KV cache as a library: each layer stores a bunch of documents (values) indexed by their summaries (keys). The query heads are different librarians, each asking a different question. There is no fundamental reason the number of librarians needs to equal the number of distinct copies of the library. As long as the librarians can ask different questions, they can share the books.

Intermediate MQA, GQA, and the bytes-per-token formula

Multi-Query Attention (MQA, Shazeer 2019)

Noam Shazeer's "Fast Transformer Decoding: One Write-Head is All You Need" made the simplest possible change: keep $h$ separate query projections, but use a single shared key projection and a single shared value projection. Every query head reads from the same K and the same V, but each head still has its own pattern of weights it learned to query with.

The cache shrinks by a factor of $h$. For Llama-1-style 64-head models, that is a 64× reduction. The catch is quality: with one set of K/V, the model loses some expressiveness. Shazeer's experiments showed a small quality hit; later work confirmed the regression was meaningful at scale.

Grouped-Query Attention (GQA, Ainslie 2023)

Joshua Ainslie and collaborators at Google found the obvious sweet spot. Pick a number of KV groups $g$ with $1 \le g \le h$. Partition the $h$ query heads into $g$ groups; each group shares one K and one V. When $g = h$ you recover MHA; when $g = 1$ you recover MQA. In practice $g = 8$ is a popular choice.

The Ainslie paper also introduced "uptraining": you start from an MHA checkpoint, mean-pool its KV heads down to $g$, then continue pretraining for ~5% of the original compute. This converted existing models to GQA cheaply, which is part of why the community adopted it so fast — Llama 2 70B shipped in July 2023, two months after the GQA paper.

The general bytes-per-token formula

Updating the formula for any of MHA / MQA / GQA, replace $h$ with the number of KV groups $g$ (which equals the number of KV heads):

$\displaystyle \text{bytes per token} = 2 \times L \times g \times d_\text{head} \times b.$

Quick sanity checks:

  • Llama-2 70B (GQA, $g=8$): $2 \times 80 \times 8 \times 128 \times 2 = 327{,}680$ bytes ≈ 320 KB / token. Sources confirm this is exactly the published number.
  • Llama-2 70B if it had stayed MHA ($g=64$): ~2.5 MB / token. GQA gave Meta an 8× memory cut for free.
  • Mistral 7B (GQA, $g=8$, $h=32$): 32 layers × 8 KV heads × 128 dims × 2 × FP16 = 131 KB / token.

Pseudocode that ties it together

# Single decode step, GQA-style
# h = num query heads, g = num KV heads, h % g == 0
def attention_step(x, kv_cache, layer):
    # Project to per-head queries and per-group K/V
    Q = x @ W_q[layer]            # shape (h, d_head)
    K_new = x @ W_k[layer]        # shape (g, d_head)
    V_new = x @ W_v[layer]        # shape (g, d_head)

    # Append to cache
    kv_cache.K[layer].append(K_new)
    kv_cache.V[layer].append(V_new)

    # Each query head attends to its group's K, V history
    out = []
    for i in range(h):
        group = i // (h // g)     # which KV group this Q head reads
        K_hist = kv_cache.K[layer][:, group]  # (T, d_head)
        V_hist = kv_cache.V[layer][:, group]
        scores = softmax(Q[i] @ K_hist.T / sqrt(d_head))
        out.append(scores @ V_hist)
    return concat(out) @ W_o[layer]

If you set $g = h$, this is MHA. If you set $g = 1$, it is MQA. The actual bytes you allocate for kv_cache.K are proportional to $g$, not $h$, which is the entire point.

Advanced Multi-Head Latent Attention and decoupled RoPE

The MLA idea: cache the latent, not the heads

Multi-Head Latent Attention, introduced in DeepSeek-V2 (Liu et al., 2024) and refined in DeepSeek-V3, asks a sharper question: instead of choosing how many K/V copies to keep, can we keep none at full width and reconstruct them on demand?

The construction is a low-rank bottleneck. From the input $h_t$ at position $t$, MLA produces:

  • A small compressed latent $c_t^{KV} \in \mathbb{R}^{d_c}$ via $c_t^{KV} = W^{DKV} h_t$, where $d_c$ is much smaller than $h \cdot d_\text{head}$.
  • A "decompression" up-projection that, when needed, recovers per-head keys $k_t^{(i)} = W_i^{UK} c_t^{KV}$ and values $v_t^{(i)} = W_i^{UV} c_t^{KV}$.

At inference time only $c_t^{KV}$ goes into the cache. The per-head K and V are recomputed each step from the latent, but recomputing them is cheap — the up-projections are small matmuls — and the cached state per token is tiny.

In DeepSeek-V2, $d_c = 512$ while the equivalent MHA state would be 128 heads × 128 = 16,384 floats. That is the source of the headline 93% reduction relative to a same-shape MHA model.

The matrix-absorption trick

The naive read of the above is that you reconstruct $k_t^{(i)}$, then compute $q_t^{(i)} \cdot k_t^{(i)}$. But the up-projection is fixed, so we can fold it into the query side:

$q_t^{(i)} \cdot k_t^{(i)} = q_t^{(i)} \cdot (W_i^{UK} c_t^{KV}) = (q_t^{(i)} W_i^{UK}) \cdot c_t^{KV}.$

That is, the query for head $i$ can be pre-multiplied by $W_i^{UK}$, and then the attention dot-product is taken directly against the cached latent $c_t^{KV}$. The up-projection never has to materialise a per-head key. This is the "matrix absorption" trick implemented in FlashInfer and SGLang. It also has a direct hardware analogue: it converts a memory-bound per-head load into a smaller, shared latent load, which is the entire point of caching it in the first place.

The RoPE problem and the decoupled fix

Rotary Position Embeddings (RoPE) inject position information by rotating each query/key in 2D subspaces by an angle proportional to the position $t$. Concretely, RoPE multiplies $q$ and $k$ by a position-dependent rotation matrix $R_t$ before the dot product.

This breaks the matrix-absorption trick. With RoPE, $k_t = R_t (W^{UK} c_t^{KV})$, and you cannot pull $W^{UK}$ through $R_t$ because the rotation is position-specific, so you cannot precompute $q W^{UK}$ once. If MLA tried to apply RoPE to its reconstructed keys, every decode step would have to re-materialise per-head keys, defeating the latent compression.

DeepSeek's decoupled RoPE solves this with a small structural addition. They split the position-bearing channels off into a separate small head:

  • A small RoPE-bearing key $k_t^R \in \mathbb{R}^{d_h^R}$ produced by a separate projection, with RoPE applied. This is shared across all heads. In V2/V3, $d_h^R = 64$.
  • A matching set of small RoPE-bearing queries $q_t^{R,(i)}$ per head, also rotated.
  • A "no-RoPE" content key $k_t^{C,(i)}$ derived from the latent $c_t^{KV}$ as before.

The attention score for head $i$ between positions $t$ and $s$ is the sum of the content term and the position term:

$q_t^{C,(i)} \cdot k_s^{C,(i)} \;+\; q_t^{R,(i)} \cdot k_s^{R}.$

The content term goes through matrix absorption against the cached latent. The position term uses a small, shared $k^R$ that is also cached but is tiny. The cache per token per layer is $d_c + d_h^R = 512 + 64 = 576$ floats. In FP8 with extra scale factors, DeepSeek-V3 stores 656 bytes per token per layer, i.e., about 70 KB across 61 layers. That is the number that beats every other dense model by a factor of five or more.

Why MLA is harder than it looks

Every other variant in this article is a one-line config change. MLA is a real architectural commitment:

  • Custom kernels. The matrix-absorption decode kernel is genuinely new code. FlashInfer added a BatchMLAPagedAttentionWrapper, vLLM and SGLang have dedicated MLA paths, and Tri Dao's FlashAttention does not natively support MLA in the same way it supports GQA. Until those kernels exist, MLA's theoretical cache savings do not translate to throughput wins.
  • Training-time complexity. The decoupled-RoPE addition adds two more projection matrices and a separate KV head with its own dimension. You cannot uptrain from an MHA checkpoint cheaply, the way GQA can be uptrained — you have to train MLA from scratch (or use an MLA-specific conversion).
  • Quality has to keep up. DeepSeek's claim — and the V2 paper supports it — is that MLA is not just smaller but actually better than MHA at matched parameter count. The intuition is that the low-rank bottleneck acts as a regulariser, much like LoRA does in fine-tuning. But this is harder to verify externally and depends on careful initialisation.

The Pope interview captures the reason it is worth all that pain: the cache size is the number that determines how many tokens you can serve from a fixed pool of HBM, and at long context the cache dominates everything. A factor of 5 reduction in KV bytes means a factor of 5 more concurrent users, or a factor of 5 longer context, on the same GPU.

The Pareto frontier

Compatibility with FlashAttention-style kernels

FlashAttention-2 and FlashAttention-3 work natively with MHA, MQA, and GQA — they treat the KV-head index as a dimension in the tile loop. GQA is a lightweight modification because the math is identical to MHA except for which K/V slot you load.

MLA is a different story. The fused kernel needs to (a) load the latent $c_s$ and shared $k_s^R$, (b) apply matrix absorption, (c) compute two dot products per head and add them, and (d) reconstruct values $v_s^{(i)}$ on the fly. Implementations now exist — FlashInfer's BatchMLAPagedAttentionWrapper and the SGLang/vLLM MLA paths — but the kernel is more specialised, and on consumer or older datacenter GPUs the fast paths sometimes regress to dense MHA-equivalent compute. This is part of the reason DeepSeek's hardware co-design is so closely watched: they target H100/H800/H20 carefully.

How this connects to API pricing

Reiner's exercise — back out cache size from Gemini's 200K crossover — works because long-context serving is bounded by KV cache, not by FLOPs. At very long contexts, the prefill is amortised, the per-step compute is constant, and the dominant memory traffic each decode step is reading the entire KV cache for the active sequences. Once cache exceeds weights, you spend more bytes per second on cache-reads than on weight-reads, which is why pricing kinks at the crossover length. A model with MLA would, all else equal, have a crossover that occurs at much longer context — directly shifting the inflection in its pricing curve.

Open questions and what to read next

  • Is MLA really better than MHA at matched parameters, or just smaller? DeepSeek's ablations say yes; independent replications are still rare.
  • Can you uptrain a GQA model into an MLA model? A few groups have tried "MHA-to-MLA" conversion; none are as plug-and-play as Ainslie's GQA uptraining recipe.
  • What replaces MLA? DeepSeek-V3.2 introduced fine-grained sparse attention on top of MLA — pruning which positions you read, on top of compressing how you store them.
  • Hybrid attention/SSM models (Mamba, Jamba, RWKV) sidestep this whole discussion by avoiding global attention. How does the cache trade-off look there?
  • How does the cache trade-off interact with speculative decoding and prefix caching? Smaller cache means more sequences fit in HBM, which compounds the wins from speculative methods.
