The 300 × Sparsity Batch-Size Rule

A back-of-the-envelope formula for the smallest inference batch that keeps the chip from idling. It comes out of one dimensionless number you can read off the spec sheet.

inference roofline mixture-of-experts economics
TL;DR

When you decode one token at a time, fetching the weights from HBM takes longer than multiplying by them. You only escape that regime by running enough sequences in parallel that the compute side catches up. Setting the two equal gives a tidy rule: the minimum efficient batch size is a hardware constant α ≈ 300 (FLOPs ÷ bandwidth in FP16) divided by the sparsity ratio. DeepSeek V3 (37 B active of 671 B) needs roughly 300 × (671 / 37) ≈ 5,400 sequences in flight before the chip is compute-bound. Providers typically run at 2-3× that to absorb real-world inefficiency. A dense Llama 70 B needs 300. Notice that nothing in the formula depends on absolute model size; only on sparsity and precision.

Why this matters

Reiner walks to the whiteboard and writes one equation: weight-fetch time equals weight-multiply time. Cancel the parameter count from both sides. On the left you're left with the chip's compute-to-bandwidth ratio. On the right you've got batch size times the active fraction of the model. Solve for batch and you have the smallest workload that keeps the chip from idling.

That one number explains a stack of otherwise mysterious facts. DeepSeek V3 felt cheap to the people who trained it and expensive to the people trying to host it. Anthropic and Google can quote prices that a startup serving the same open weights can't match. MoE only pays off if you have demand to fill it. Output tokens cost three to five times input tokens. This is the cleanest entry point into the roofline picture; if you've read tutorial 01, you already have what you need.

The goal for this page: given a new chip's spec sheet and a new model's parameter counts, work out in your head what batch size you need to serve it profitably and how many concurrent users that implies.

Beginner arithmetic intensity, recapped

The two clocks running in parallel

When a graphics processing unit (GPU) or tensor processing unit (TPU) executes a matrix multiplication, two physical processes happen at once. Operands move from off-chip HBM into on-chip registers, and the tensor cores multiply and accumulate them. Each has a fixed maximum rate: memory bandwidth (bytes per second) and compute throughput (FLOPs per second, where FLOP is a single floating-point operation). The wall-clock time of the operation is whichever process takes longer.

Roofline analysis is just the observation that you can plot, on the same axes, the rate of compute as a function of arithmetic intensity (the number of FLOPs you do per byte you fetch). Below a critical intensity you are bandwidth-bound; above it you are compute-bound. The kink in the curve sits exactly where memory time equals compute time. We covered the full picture in tutorial 01; here we just need one of its consequences.

The hardware constant α

Take any modern accelerator and divide its peak FLOP rate (in 16-bit floating point, or FP16) by its HBM bandwidth (in bytes per second). The dimensions cancel except for FLOPs per byte, which is a unitless count once you fix the precision. For an NVIDIA H100 SXM running BF16 tensor cores: 989 TFLOP/s ÷ 3.35 TB/s ≈ 295. For an H200: 989 / 4.8 ≈ 206. For a B200: roughly 2,250 / 7.7 ≈ 290. For TPU v5e: 197 / 0.82 ≈ 240. Across a decade of accelerators the answer keeps falling between roughly 200 and 400 in FP16. Reiner rounds this to 300 and calls it a hardware constant.

Why is it stable?

Both quantities have grown roughly together for a decade. Each new HBM generation roughly doubles bandwidth; each new tensor-core generation roughly doubles FLOPs. Vendors do not optimise α directly, but they do balance the chip area they spend on stream multiprocessors against the area they spend on memory controllers, and the equilibrium has not moved much. Memory cells are constrained by physics (capacitor leakage, IO pitch); transistor counts are constrained by Moore's law. They scale at similar rates.

What does change α is precision. Drop from FP16 to FP8 and you halve the bytes per parameter without halving the FLOP rate (FP8 tensor cores are roughly 2× faster than FP16 on Hopper and Blackwell), so α rises to ~600. Drop to INT4 with FP16 accumulation and it climbs further. Quantisation, in this picture, is just a way to push α upward and therefore push the required batch size upward too — cheaper compute per byte means you have to do more compute to balance the byte transfers.

What "batch size" means in decoding

During training, batch size is unambiguous: it is the number of independent training examples processed in one forward-backward pass. During inference, prefill (processing the prompt) is similar — the prompt's tokens all go through together. But during decoding, the autoregressive nature of language models forces you to generate tokens one at a time per sequence. So "batch size" during decoding means the number of distinct user sequences you advance simultaneously. If 64 users are all waiting for their next token, the GPU can multiply by the model weights once and use the result for all 64 sequences. Each load of weights from HBM amortises across 64 multiplications instead of one. That is the only lever you have to make decoding faster per dollar.

Intermediate the derivation, KV cache, and worked examples

The two-line derivation

Let $N_\text{total}$ be the total number of model parameters and $N_\text{active}$ the number of parameters used per token (in a dense model these are equal; in an MoE they differ). Let $b$ be the number of bytes per parameter (2 for BF16, 1 for FP8, 0.5 for INT4). Let $B$ be the decoding batch size in sequences, $\beta$ the chip's HBM bandwidth in bytes/second, and $\phi$ its compute throughput in FLOPs/second.

Per generated token, two things have to happen on the chip:

  • Fetch every weight that will be used into registers. Critically, you must load all $N_\text{total}$ parameters because in MoE models you cannot know in advance which experts each sequence will route to (and the experts are sharded across the same HBM either way). Time = $N_\text{total} \cdot b / \beta$.
  • Multiply each active weight by each sequence's activation. Each fused multiply-add is 2 FLOPs. Total FLOPs per token, across the batch, = $2 \cdot B \cdot N_\text{active}$. Time = $2 \cdot B \cdot N_\text{active} / \phi$.

The optimal batch sits where neither dominates. Set the two times equal:

$$\frac{N_\text{total} \cdot b}{\beta} = \frac{2 \cdot B \cdot N_\text{active}}{\phi}$$

Solve for $B$:

$$B^{\star} = \frac{\phi}{2\beta / b} \cdot \frac{N_\text{total}}{N_\text{active}} = \frac{\phi \cdot b}{2 \beta} \cdot \frac{1}{s}$$

where $s = N_\text{active} / N_\text{total}$ is the sparsity ratio (in a dense model, $s = 1$). The first factor is the FLOPs-per-byte of the chip at the given precision — that is our friend α. (The factor of 2 comes from FMA being 2 FLOPs; some sources quote a slightly different constant. Either way it is absorbed into the "~300" approximation.) So:

$$\boxed{\; B^{\star} \;\approx\; \alpha \cdot \frac{N_\text{total}}{N_\text{active}} \;\approx\; 300 \cdot \frac{1}{s} \;}$$

That is the entire rule. Below $B^{\star}$ you are throwing away tensor-core cycles waiting for HBM. Above it you are throwing away HBM bandwidth waiting for tensor cores. Both are forms of leaving money on the table; the first is far more common in practice because real workloads do not arrive in batches of thousands without effort.

Worked example 1: a dense 70 B model on H100

Llama 3.3 70 B is dense, so $s = 1$ and $B^{\star} \approx \alpha$. On an H100 SXM in BF16:

  • $\phi = 989$ TFLOP/s; $\beta = 3.35$ TB/s; $b = 2$ bytes
  • $\alpha = \phi \cdot b / (2\beta) = (989 \times 10^{12} \cdot 2) / (2 \cdot 3.35 \times 10^{12}) \approx 295$

So the kink sits at ~295 concurrent sequences. The whole 70 B model in BF16 needs 140 GB; that does not fit on an 80 GB H100, so in practice you shard across two H100s, but the per-chip arithmetic stays the same because both the weight bytes and the FLOPs scale together. Drop to FP8 (which Llama 3.3 70 B inference servers commonly use): $\phi$ doubles to 1,979 TFLOP/s and $b$ halves to 1, so $\alpha$ goes from 295 to ~590. The kink sits at ~590 sequences. Quantisation makes the chip cheaper per token but raises the bar for batch.

Worked example 2: DeepSeek V3 on a B200 rack

DeepSeek V3 has 671 B total parameters and 37 B active per token (256 routed experts, 8 selected, plus a shared expert and dense components). Sparsity:

$s = 37 / 671 \approx 0.055$, so $1/s \approx 18.1$

On a B200 in FP8, $\alpha \approx 600$. Plug in:

$B^{\star} \approx 600 \cdot 18.1 \approx 10{,}900$ concurrent sequences

In BF16 (where $\alpha \approx 290$) the answer is closer to 5,300. Reiner uses the FP16 number and the rougher "32 of 256 experts" sparsity in the interview, getting an answer of ~2,400 sequences; the calculation is sensitive to exactly how you count the dense components. The three commonly quoted numbers in the literature for DeepSeek V3's optimal batch — 2,400, ~3,000, and ~5,400 — differ by only a small constant factor. The headline conclusion is the same: tens of thousands of times the batch you want for a dense model of similar quality. That is the cost of the MoE architecture's inference profile, and it is exactly why DeepSeek's papers report the model is cheap when you can fill the batch but expensive when you cannot.

Two facts jump out. First, the rule is scale-invariant: only the ratio matters, not the absolute parameter count. A 7 B dense model and a 700 B dense model have the same $B^{\star}$. Second, the rule is quantisation-sensitive: it scales with $\alpha$, which roughly doubles each time you halve precision.

The KV cache changes the equation at long context

So far we ignored attention. Each layer's attention reads, for each sequence, that sequence's key-value (KV) cache from HBM. Unlike the model weights, the KV cache cannot be amortised across the batch: each user has their own. So adding the cache term to the memory side gives:

$$\text{memory time} = \frac{N_\text{total} \cdot b + B \cdot K(L) \cdot b}{\beta}$$

where $K(L)$ is the per-sequence cache size at context length $L$. Setting equal to compute time and solving:

$$B^{\star} \approx \alpha \cdot \frac{N_\text{total}}{N_\text{active} - \alpha \cdot K(L)}$$

For short contexts, $\alpha \cdot K(L) \ll N_\text{active}$ and the formula reduces to the simple version. As $L$ grows, the denominator shrinks; once $\alpha \cdot K(L)$ approaches $N_\text{active}$, no batch size is large enough to make compute catch up — you are pinned to the memory roof forever. This is one reason long-context generation is so expensive. Modern designs (multi-head latent attention in DeepSeek, grouped-query attention in Llama) explicitly shrink $K(L)$ to keep the formula well-behaved out to 128k tokens.

Why providers run 2-3× past the balance point

The roofline is an idealisation: it assumes perfect overlap of memory and compute, perfect pipeline scheduling, and zero overhead for routing in MoE models. Real systems lose 20-40% to bubbles, all-to-all communication, expert imbalance, and stragglers. A common operational rule is to size for 2-3× the analytical $B^{\star}$ so that the chip is comfortably compute-bound even when the empirical curves bend earlier than theory predicts. Pope says it directly in the interview:

"Generally, people will go a little bit larger than this. They don't really want to be exactly at the balance point because real-world efficiencies aren't as good as a roofline analysis would say."

The penalty for being a bit too high is small (a small fraction of memory bandwidth wasted); the penalty for being a bit too low is brutal (compute idles, your dollars-per-token shoot up). When the cost surface is asymmetric, prudent engineers err on the safe side.

Advanced latency floors, the 20 ms train, and who can serve frontier models

The latency floor: capacity over bandwidth

Even with infinite users, there is a hard floor on per-token latency. The chip must read the entire model from HBM once per generated token, and that takes:

$T_\text{floor} = \dfrac{\text{HBM capacity used by the model}}{\text{HBM bandwidth}}$

For an H100 holding 70 B parameters in BF16 (140 GB across two cards) at 3.35 TB/s per card, that is ~21 ms. For a Blackwell B200 with 180 GB at 7.7 TB/s, the bound is ~23 ms if the model fully fills HBM. Pope notes that across many HBM generations the answer keeps coming out near 20 ms because, again, capacity and bandwidth grow together. From the interview:

"For example, on the Rubin generation, it is something like 288 gigabytes divided by 20 terabytes per second. This comes out to about 15 milliseconds."

The train-leaves-every-20-ms metaphor

If you are running with a fixed batch size $B$, your inference server is essentially a station from which a train departs every 20 ms. Each "train" is one weight-pass through HBM, and it carries up to $B$ passengers (sequences) at no marginal cost. A user sequence arrives, waits at most 20 ms for the next train, rides one stop (gets one new token), and gets back in line. Worst-case end-to-end token latency is therefore ~40 ms (up to 20 ms wait + 20 ms ride).

Throughput in tokens per second

If trains depart every $T_\text{floor}$ seconds and carry $B$ passengers, the total throughput is $B / T_\text{floor}$ tokens per second per chip. For a frontier MoE on a Blackwell rack of 72 GPUs, a typical headline number Pope quotes is:

$\text{throughput} \approx B^{\star} \cdot 64 \approx 2{,}000 \cdot 64 = 128{,}000$ tokens/s per rack

That number is the entry ticket. If you cannot maintain at least 128,000 tokens/s of demand for a frontier model, your rack runs below $B^{\star}$ and your cost-per-token explodes. Anthropic, OpenAI, Google, and Meta easily run hundreds of millions of tokens per second worldwide. A lab with one rack and modest inbound traffic does not, which is why hosting frontier MoEs is structurally a centralised business. From the interview:

"Gemini last year was in the hundreds of millions of tokens per second worldwide. This is one-thousandth of that."

Cost-per-token vs batch size

Combining everything, the chip-cost-per-token has two regimes. For $B < B^{\star}$, you waste tensor-core cycles, so cost falls roughly as $1/B$. Once $B \ge B^{\star}$, you saturate the compute roof and cost stops falling. The curve is therefore an L-shape with the elbow at $B^{\star}$. Latency, on the other hand, stays at $T_\text{floor}$ throughout and only worsens once $B$ pushes past saturation (when compute starts to dominate). The two curves together explain the operating sweet spot: get $B$ to a couple of times $B^{\star}$, sit there, and keep the trains full.

Why scale falls out of the equation

Notice that nothing in $B^{\star} = \alpha / s$ involves $N_\text{total}$. Doubling the model's parameter count doubles both the weight-fetch time and the available FLOPs at a given batch — the crossover does not move. What does move is the latency floor (because $T_\text{floor}$ scales with model size) and the absolute throughput (which still equals $B^{\star} / T_\text{floor}$). So bigger models are slower per token but no more demanding in batch size, all else equal. The constraint that determines who can profitably serve a model is sparsity, not scale.

Sparsity as the new lever

If MoE architectures keep getting sparser — some research models propose 1:64 or 1:128 active-to-total — the required batch grows linearly. A 1:64-sparse model on a B200 needs ~38,000 concurrent sequences. The chip-cost-per-token at that batch is excellent, but only if you can actually fill the batch. Reiner's argument in the interview is that this dynamic structurally favours the largest providers: the more aggressively a frontier model exploits sparsity, the more demand you need to serve it economically, and only a handful of companies have that demand. The rule is not just an engineering observation; it is a market-structure prediction.

Open questions and what to read next

  • How does the rule change for speculative decoding, where each "step" produces multiple tokens at the cost of more compute? In effect, speculative decoding raises the FLOPs side of the balance without raising the bytes side, lowering $B^{\star}$.
  • The KV-cache term assumes uniform context length. In practice, requests have wildly different lengths and providers use continuous batching. What is the right effective $B^{\star}$ when the workload is non-uniform?
  • If 3D-stacked DRAM (HBM4, HBM-PIM, near-memory compute) shifts the FLOP-per-byte ratio in either direction, how does that reshape model architecture choices?
  • Are training-only sparsity techniques (such as block-sparse attention) easier to serve than MoE sparsity, because the blocks are static and predictable?
  • How does pipeline parallelism across multiple racks interact with the rule? A single rack saturates near $B^{\star}$, but cross-rack traffic introduces new bandwidth-limited steps.
